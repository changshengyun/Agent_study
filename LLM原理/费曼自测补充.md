# 2. Token、Token ID 和 Token Embedding 的修正

## 2.1 Token

Token 是原始文本经过 Tokenizer 切分后得到的离散符号。

例如：

```text
原始文本：
我喜欢 Agent

Token：
["我", "喜欢", " Agent"]
```

Token 仍然是文本符号，不是数字。

Token 可能是：

* 一个汉字；
* 一个英文单词；
* 单词的一部分；
* 标点；
* 空格和单词的组合；
* 特殊控制符号。

---

## 2.2 Token ID

每一个 Token 都会在模型词表中对应一个整数编号。

例如：

```text
"我"       → 105
"喜欢"     → 872
" Agent"   → 31025
```

因此：

```text
Token
→ 查询模型词表
→ Token ID
```

Token ID 只是索引，不代表语义大小。

例如 Token ID 30000 并不比 Token ID 100 更重要，也不意味着它的语义更复杂。

---

## 2.3 Token Embedding

LLM 内部存在一个可训练的 Embedding Matrix。

Token ID 用来查询该矩阵中的某一行：

```text
Token ID
→ 查询 Embedding Matrix
→ Token Embedding
```

Token Embedding 是一个浮点向量，例如：

```text
[0.12, -0.31, 0.56, ...]
```

它是 Transformer 可以进行矩阵运算的初始表示。

因此三个概念的数据类型分别是：

```text
Token：文本符号
Token ID：整数索引
Token Embedding：浮点向量
```

---

## 2.4 Token Embedding 与独立 Embedding Model

二者不能混淆。

| 对象              | 输入          | 输出             | 作用                      |
| --------------- | ----------- | -------------- | ----------------------- |
| Token Embedding | 一个 Token ID | 单个 Token 的内部向量 | 供 LLM 内部 Transformer 计算 |
| Embedding Model | 一段文本、图片等    | 整体语义向量         | RAG、相似度检索、聚类            |

独立 Embedding Model 并不是简单地“位于 LLM 外部”，它与 Token Embedding 最本质的区别是：

> 两者承担的任务目标不同。

# 3. 为什么模型不能直接计算文本

神经网络内部主要执行：

* 矩阵乘法；
* 向量加法；
* 归一化；
* 非线性变换；
* 概率计算。

它无法直接对字符串“我”“喜欢”“Agent”进行数学运算。

所以需要进行：

```text
字符串
→ Token
→ Token ID
→ 浮点向量
→ 神经网络计算
```

模型内部从输入到输出处理的主要对象都是数字张量。

模型最后也不是直接生成文本，而是：

```text
Logits
→ 选择 Token ID
→ Detokenization
→ 文本
```

# 4. 训练并行与推理串行

## 4.1 推理为什么逐 Token 生成

推理阶段未来 Token 还不存在。

例如当前输入：

```text
人工智能正在
```

模型必须先生成：

```text
改变
```

然后才能基于：

```text
人工智能正在改变
```

继续预测：

```text
世界
```

后一个 Token 依赖前一个实际生成出的 Token，因此存在顺序依赖：

```text
Token t
→ 作为新条件
→ 生成 Token t+1
```

这就是自回归 Decode 通常必须串行执行的原因。

需要特别纠正：

> 逐 Token 生成不是因为 API 开启了 Streaming。

即使关闭 Streaming，模型内部仍然逐 Token 生成，只是服务端等全部生成完后再一次返回。

---

## 4.2 训练为什么能够并行

训练时完整文本已经提前存在。

例如：

```text
人工智能正在改变世界
```

可以同时构造以下训练任务：

```text
看到“人工智能”         → 预测“正在”
看到“人工智能正在”     → 预测“改变”
看到“人工智能正在改变” → 预测“世界”
```

通过 Causal Mask，每个位置只能看到自己和前面的 Token，不能读取未来答案。

但因为完整序列已经存在，Transformer 可以在一次矩阵计算中同时处理多个位置。

因此完整原因是：

```text
训练：
完整目标序列已知
+ Causal Mask 防止偷看未来
+ Transformer 支持位置并行计算

推理：
未来 Token 未知
+ 后一个 Token 依赖前一个实际采样结果
→ 必须逐步生成
```

# 5. Temperature、Top-p 与 Top-k

## 5.1 Temperature

Temperature 作用于 Logits 到概率分布的转换阶段。

基本链路：

```text
LM Head
→ Logits
→ Temperature 调整
→ Softmax
→ 概率分布
```

Temperature 较低时：

* Logits 差距被放大；
* 概率分布更尖锐；
* 高概率 Token 更突出；
* 输出通常更稳定。

Temperature 较高时：

* Logits 差距被缩小；
* 概率分布更平坦；
* 低概率 Token 更容易被选择；
* 输出通常更多样。

Temperature 控制的是：

> 候选 Token 之间的概率差距。

它不直接规定候选 Token 的数量。

---

## 5.2 Top-p

Top-p 是累计概率采样，也称 Nucleus Sampling。

假设概率为：

```text
A：0.45
B：0.30
C：0.15
D：0.07
E：0.03
```

当：

```text
top_p = 0.8
```

从高到低累加：

```text
A          = 0.45
A + B      = 0.75
A + B + C  = 0.90
```

加入 C 后第一次达到或超过 0.8，因此候选集合为：

```text
{A, B, C}
```

系统再对候选集合重新归一化，并采样出一个 Token。

Top-p 不是：

```text
只选择单个概率大于 0.8 的 Token
```

---

## 5.3 Top-k

Top-k 固定保留概率最高的前 (k) 个 Token。

例如：

```text
top_k = 5
```

表示只从概率最高的五个候选 Token 中进行采样。

区别如下：

| 参数          | 作用                 |
| ----------- | ------------------ |
| Temperature | 调整概率分布的尖锐程度        |
| Top-k       | 固定保留前 k 个候选        |
| Top-p       | 保留累计概率达到 p 的最小候选集合 |

# 6. Structured Output 的保证边界

Structured Output 主要解决：

> 模型输出能否被程序稳定解析。

它可以约束：

* 输出为合法 JSON；
* 必填字段存在；
* 字段类型正确；
* 枚举值符合要求；
* 输出符合 JSON Schema。

例如：

```json
{
  "task_type": "search",
  "priority": "high"
}
```

程序可以直接读取字段，而不必从自然语言中解析。

但是 Structured Output 不能保证：

* 内容事实正确；
* 数字计算正确；
* 工具参数合理；
* 业务判断正确；
* 工具已经执行；
* 用户目标已经完成。

例如：

```json
{
  "email_sent": true
}
```

结构完全正确，但这可能只是模型生成的布尔值，不能证明邮件真的发送。

因此：

```text
Structured Output
→ 解决可解析性

Validator
→ 检查正确性

Tool / Runtime
→ 完成真实执行
```

需要记住：

```text
JSON 正确
≠ 语义正确
≠ 事实正确
≠ 执行成功
```

# 7. Streaming 的准确理解

Streaming 不改变模型内部的自回归生成机制。

## 非流式

```text
模型生成完整回答
→ 服务端收集完整结果
→ 一次性返回客户端
```

## 流式

```text
模型生成部分内容
→ 服务端立即返回一个 Chunk
→ 继续生成
→ 继续返回
```

网络中的 Chunk 不一定等于一个 Token。

一个 Chunk 可能包含：

* 一个 Token；
* 多个 Token；
* 一个文本片段。

因此：

```text
Token：模型内部生成单位
Chunk：网络传输单位
```

Streaming 主要改善用户感知到的首字等待时间，但不会显著降低：

* 模型总计算量；
* 输出 Token 数量；
* Token 费用；
* 完整回答的总生成时间。

# 8. Prefill 与 Decode

## 8.1 Prefill

Prefill 是模型处理全部输入上下文的计算阶段。

输入可能包括：

```text
System Prompt
+ 用户输入
+ 历史对话
+ RAG 内容
+ 工具定义
+ 工具返回结果
```

Prefill 阶段会：

1. 处理所有输入 Token；
2. 经过 Transformer；
3. 计算 Hidden State；
4. 建立初始 KV Cache；
5. 得到第一个输出 Token 的 Logits。

长输入主要增加：

* Prefill 计算量；
* TTFT；
* 输入 Token 费用；
* KV Cache 初始占用；
* 显存压力。

---

## 8.2 Decode

Decode 是逐 Token 生成输出的阶段。

每生成一个新 Token，都需要：

```text
新 Token Embedding
→ 经过全部 Transformer 层
→ 读取历史 KV Cache
→ 生成 Logits
→ 采样下一个 Token
```

长输出主要增加：

* Decode 次数；
* 总生成时间；
* 输出 Token 费用；
* 后续对话的输入成本。

---

## 8.3 TTFT 与总延迟

TTFT 是从请求发出到收到第一个 Token 的时间。

主要受：

* 请求排队；
* 网络延迟；
* 输入长度；
* Prefill；
* 模型大小；
* GPU 性能；
* Prompt Cache；

影响。

总延迟可以粗略理解为：

```text
总延迟
=
排队和网络
+ Prefill
+ Decode
+ 工具调用
+ 重试
+ Agent 多轮循环
```

因此：

```text
长输入主要影响 Prefill 和 TTFT
长输出主要影响 Decode 和总延迟
```

# 9. KV Cache 的关键修正

这是费曼自测中最重要的错误点。

## 9.1 KV Cache 保存什么

KV Cache 保存的是：

> 历史 Token 在 Transformer 每一层 Attention 中已经计算出的 Key 向量和 Value 向量。

它不是保存：

* Attention 权重；
* 完整 Attention 矩阵；
* Token 对应的最终答案；
* 某种 Token 命中记录。

其结构可以理解为：

```text
每一个 Transformer 层
× 每一个历史 Token
× Key 向量
× Value 向量
```

---

## 9.2 为什么能够加速生成

假设历史序列已经是：

```text
人工智能正在改变
```

这些历史 Token 在每一层中的 K 和 V 已经计算过。

当模型生成下一个 Token 时，只需要：

1. 计算新 Token 的 Q、K、V；
2. 用新 Token 的 Q 读取历史 K；
3. 根据 Attention 权重聚合历史 V；
4. 将新 Token 的 K、V 追加到缓存。

因此：

```text
不是某个 Token“命中缓存”
而是复用了所有历史 Token 已计算好的 K/V
```

KV Cache 避免了每生成一个 Token 时，将所有历史 Token 再次完整通过模型计算。

---

## 9.3 KV Cache 为什么占用大量显存

KV Cache 占用大致与以下因素成正比：

```text
序列长度
× Transformer 层数
× KV Head 数
× Head Dimension
× K 和 V 两组数据
× 数值精度
× 并发请求数
```

因此：

> KV Cache 存储空间通常随上下文长度近似线性增长，即 O(n)。

需要区别：

| 对象                       | 随序列长度增长        |
| ------------------------ | -------------- |
| Prefill 中标准 Attention 计算 | 可能接近平方增长 O(n²) |
| KV Cache 存储空间            | 通常线性增长 O(n)    |

之前将 KV Cache 显存理解为 Token 数量平方，是把 Attention 计算复杂度和缓存存储复杂度混淆了。

# 10. 模型为什么能够专业地说错话

LLM 的核心训练目标主要是：

```text
根据当前上下文
→ 生成概率较高的下一个 Token
```

而不是：

```text
连接现实世界
→ 自动验证每一个事实
```

模型学习了大量：

* 论文表达格式；
* 专业术语；
* 代码结构；
* 引用格式；
* 推理语言；
* 正式写作风格。

因此即使不知道真实答案，也可能生成：

* 看似真实的论文名称；
* 看似合理的作者；
* 格式完整的 API 参数；
* 结构专业的错误推理。

所以：

```text
语言合理性
≠ 事实真实性
```

模型可能在“语言生成”任务上表现成功，却在“事实验证”任务上失败。

---

# 11. 为什么 Temperature 为 0 仍然会幻觉

Temperature 控制的是采样随机性，不会修改模型参数、知识或推理过程。

假设模型概率为：

```text
错误答案：80%
正确答案：15%
不知道：5%
```

较低 Temperature 会让模型更加稳定地选择最高概率的错误答案。

因此：

```text
低 Temperature
= 输出更稳定

低 Temperature
≠ 事实更加正确
```

幻觉的解决方案不能只依赖降低 Temperature。

---

# 12. 为什么 RAG 不能彻底消除幻觉

RAG 给模型提供外部证据，但整个链路仍然存在多个可能出错的环节：

```text
用户问题理解错误
→ 查询改写错误
→ 文档库内容错误或过时
→ 文档切片丢失上下文
→ 召回错误
→ Rerank 排序错误
→ 关键内容被截断
→ LLM 忽略证据
→ LLM 错误解释证据
→ LLM 补充证据中不存在的信息
```

因此 RAG 的准确作用是：

> 降低模型仅依赖参数知识回答的风险，并为回答提供可验证证据。

RAG 不是自动事实保证系统。

# 13. 模型边界的具体含义

模型边界用于决定：

> 在一个系统中，哪些责任可以交给 LLM，哪些责任必须交给工具、代码、数据库、Runtime、Validator 或用户。

它主要服务于：

1. 系统设计；
2. 风险控制；
3. 故障定位；
4. 验收标准设计。

---

## 13.1 知识边界

LLM 不保证：

* 知道所有事实；
* 知识始终最新；
* 参数知识始终准确。

例如查询今天的股票价格，必须调用实时数据 API。

接管组件：

```text
搜索工具
数据库
RAG
官方 API
```

---

## 13.2 观察边界

LLM 不能直接看到真实环境。

它不能天然知道：

* 文件是否存在；
* 服务是否启动；
* 数据库当前状态；
* 邮件是否送达；
* 代码是否运行成功。

需要：

```text
真实环境
→ Tool 读取
→ Observation
→ 放入 Context
→ LLM 分析
```

接管组件：

```text
工具
监控系统
数据库查询
文件系统 API
```

---

## 13.3 行动边界

LLM 输出的本质是 Token。

例如模型输出：

```json
{
  "action": "send_email"
}
```

这只是动作请求，不代表邮件已经发送。

真实执行流程：

```text
LLM 提出动作
→ Runtime 校验
→ 权限判断
→ Tool 执行
→ 外部系统改变
```

接管组件：

```text
Agent Runtime
权限系统
Tool Executor
外部 API
```

---

## 13.4 验证边界

LLM 不能仅靠自己的声明证明结果正确。

例如：

```text
模型说代码正确
≠ 代码编译通过
```

需要：

```text
编译
单元测试
API 状态查询
数据库回查
文件检查
规则校验
人工审核
```

接管组件：

```text
Validator
测试系统
规则代码
人工审批
```

---

# 14. Validator 是什么

Validator 是负责检查结果是否满足要求的组件。

它不一定是另一个 LLM。

Validator 可以是：

* JSON Schema；
* 参数规则；
* 文件存在性检查；
* 数据库回查；
* 单元测试；
* API 状态检查；
* 第二个模型；
* 人工审核。

原则是：

> 对确定性事实，应优先使用确定性方法进行验证。

例如创建文件的验收不应只是：

```text
create_file API 返回 success
```

而应该检查：

```text
文件存在
+ 路径正确
+ 文件非空
+ 内容符合要求
```

# 15. “查找三篇最新论文并发送给导师”的职责划分

## 第一步：LLM 理解任务

LLM 负责提取：

* 论文主题；
* 数量；
* “最新”的时间标准；
* 总结要求；
* 邮件格式。

---

## 第二步：搜索工具获取真实论文

搜索工具或论文数据库负责返回：

* 真实标题；
* 作者；
* 发布时间；
* 摘要；
* 原文链接。

LLM 不应仅凭参数记忆编造最新论文。

---

## 第三步：Validator 验证论文

检查：

* 是否确实有三篇；
* 是否与主题相关；
* 是否按照日期排序；
* 标题、作者、链接是否一致；
* 链接是否可以访问。

---

## 第四步：LLM 总结论文并生成邮件草稿

LLM 负责：

* 总结方法和结论；
* 比较论文差异；
* 组织邮件内容；
* 调整语言风格。

---

## 第五步：用户审批

因为发送邮件会改变外部状态，通常需要用户确认：

* 收件人；
* 邮件主题；
* 正文；
* 附件。

---

## 第六步：Runtime 调用邮件工具

Runtime 负责：

* 校验邮箱；
* 调用邮件 API；
* 处理超时；
* 记录执行状态；
* 防止重复发送。

---

## 第七步：Validator 验证发送结果

检查：

* API 是否成功；
* 已发送记录中是否存在；
* 收件人是否正确；
* 附件是否完整；
* 邮件内容是否符合要求。

---

# 16. 三个“完成”不能画等号

```text
模型说完成
≠
工具执行成功
≠
用户目标完成
```

## 模型说完成

只是模型生成了一句话。

## 工具执行成功

只说明某个技术操作成功。

例如邮件 API 成功，但可能：

* 收件人错误；
* 正文错误；
* 缺少附件。

## 用户目标完成

必须验证最终状态是否满足用户真实需求。

因此正确层级是：

```text
模型声明完成
→ 工具状态验证
→ 业务后置条件验证
→ 用户目标完成
```

---

# 17. Agent 重复调用工具的原因

Agent 不断调用同一个工具，可能来自多个层级。

| 层级                    | 可能问题              |
| --------------------- | ----------------- |
| LLM                   | 错误判断仍然需要调用工具      |
| Prompt                | 没说明什么时候停止         |
| State                 | 没记录工具已经执行         |
| Runtime               | 没有重复调用检测          |
| Termination Condition | 没定义任务完成条件         |
| Tool Result           | 返回信息不清晰，模型不知道是否成功 |

例如工具只返回：

```text
success
```

但没有返回实际检索内容，模型可能认为任务没有获得结果，于是再次调用。

Runtime 应增加：

* 最大迭代次数；
* 相同工具和参数去重；
* 工具调用预算；
* 超时；
* 死循环检测；
* 幂等机制；
* 明确终止条件。

---

# 18. 幂等性

幂等性表示：

> 同一个操作重复执行多次，不应该产生额外的重复副作用。

例如 Agent 因超时重试发送邮件，如果没有幂等机制，可能重复发送三封邮件。

常见解决方式：

* Idempotency Key；
* 执行记录；
* 唯一任务 ID；
* 数据库去重；
* 操作前查询状态。

这是 Agent 执行层的重要工程约束。

LLM 负责想，

# 23. 后续复习时只需抓住的六句话

1. Token 是文本单位，Token ID 是整数索引，Token Embedding 是浮点向量。
2. 推理逐 Token 生成是因为未来 Token 未知，不是因为 Streaming。
3. Temperature 调整概率形状，Top-k 和 Top-p 筛选候选集合。
4. Structured Output 保证可解析，不保证事实和业务正确。
5. KV Cache 保存历史 Token 在每层的 K/V，存储通常随上下文长度线性增长。
6. LLM 负责候选判断，Runtime 控制流程，Tool 真实执行，Validator 验证结果。
