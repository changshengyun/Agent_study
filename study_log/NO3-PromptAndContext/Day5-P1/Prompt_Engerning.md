# 2. Prompt Engineering 是什么

Prompt Engineering 是通过编写、调整和测试 Prompt，使模型输出更加符合业务需求的过程。

它不是单纯地“把问题写得更详细”，而是持续完成以下循环：

```text
编写 Prompt
→ 调用模型
→ 检查输出
→ 发现问题
→ 修改 Prompt
→ 重新测试
```

Prompt Engineering 的最终目标是：

> 让模型在指定输入条件下，稳定地产生满足业务目标的结果。

# 3. System Prompt

## 3.1 System Prompt 的作用

System Prompt 用于设定模型的：

* 身份；
* 职责；
* 沟通风格；
* 总体目标；
* 执行规则；
* 禁止行为；
* 输出要求。

例如：

```text
你是一名论文阅读助手，负责帮助用户阅读和总结英文论文。
```

System Prompt 会作为模型当前上下文中的高优先级指令，但它并不是绝对安全边界。

---

## 3.2 System Prompt 的添加方式

### 方法一：在 messages 中添加 `SystemMessage`

```python
messages = [
    SystemMessage(content="你是一名论文阅读助手。"),
    HumanMessage(content="总结这篇论文的摘要。"),
]
```

消息结构中常见的角色包括：

```text
SystemMessage：系统规则
HumanMessage：用户输入
AIMessage：模型之前的回答
ToolMessage：工具返回结果
```

### 方法二：在 Model 或 Agent 中设置 `system_prompt`

```python
system_prompt = "你是一名论文阅读助手。"
```

然后在创建 Agent 或调用模型时传入。

两种方式的核心目的相同：

> 把稳定的系统级任务规则放入模型上下文。

# 4. Prompt 的主要组成

一个较完整的 Prompt 可以包含以下部分。

## 4.1 Identity：身份

Identity 用于描述模型的：

* 角色；
* 职责；
* 沟通方式；
* 总体任务方向。

例如：

```text
你是一名计算机科学论文阅读助手。
```

---

## 4.2 Instructions：指令

Instructions 用于说明模型应该怎样执行任务。

包括：

* 应该做什么；
* 不应该做什么；
* 处理顺序；
* 输出要求；
* 证据要求。

例如：

```text
请提取论文的研究问题、方法、实验结果和局限。
不得补充论文中不存在的信息。
```

---

## 4.3 Examples：示例

示例用于向模型展示：

* 可能的用户输入；
* 期望的模型输出；
* 正确的格式；
* 正确的判断方式。

例如：

```text
User：
总结这篇论文的摘要。

Assistant：
## 研究问题
……

## 核心方法
……
```

示例的价值不只是展示格式，还可以展示模型应如何处理特殊情况。

---

## 4.4 Context：背景信息

Context 是模型完成当前任务所需要使用的材料，例如：

* 论文正文；
* RAG 检索结果；
* 用户笔记；
* 历史对话；
* 工具返回结果；
* 业务背景数据。

需要注意：

> Context 是模型执行任务的依据，而不是任务规则本身。

---

# 5. Prompt 的表达格式

## 5.1 Markdown

推荐使用 Markdown 组织 Prompt。

例如：

```markdown
# 身份

你是一名论文阅读助手。

# 目标

生成一份结构化论文总结。

# 规则

- 只使用论文原文；
- 不得编造数据；
- 缺少证据时明确说明。
```

Markdown 的优点是：

* 层级清晰；
* 便于阅读；
* 便于维护；
* 适合长 Prompt。

---

## 5.2 XML 标签

XML 标签可以帮助区分不同内容块。

例如：

```xml
<instructions>
只根据论文原文回答。
</instructions>

<paper>
这里放论文内容。
</paper>

<user_question>
论文的核心方法是什么？
</user_question>
```

它适合用于明确隔离：

* 指令；
* 数据；
* 用户问题；
* 示例；
* 工具结果。

---

## 5.3 JSON

JSON 适合表示结构化数据。

### JSON 对象

```json
{
  "name": "Tim"
}
```

Python 中对应：

```python
dict
```

### JSON 数组

```json
[
  {
    "name": "Tim"
  },
  {
    "name": "Alice"
  }
]
```

Python 中对应：

```python
list[dict]
```

### Python 与 JSON 的转换

```python
json.dumps()
```

用于：

```text
Python 对象 → JSON 字符串
```

```python
json.loads()
```

用于：

```text
JSON 字符串 → Python 对象
```

# 6. 一个基础 System Prompt 示例

```python
system_prompt = """
# 身份

你是一名论文阅读助手，负责帮助用户阅读英文论文。

# 指令

- 请使用 Markdown 格式分点输出；
- 只根据用户提供的论文内容回答；
- 不得补充原文中不存在的事实；
- 证据不足时明确说明。

# 示例

User：
总结论文的摘要部分。

Assistant：
## 研究问题
……

## 核心方法
……
"""
```

这个 Prompt 已经包含：

* 身份；
* 执行规则；
* 禁止行为；
* 输出格式；
* 示例。

---

# 7. Zero-shot

Zero-shot 是不给模型提供输出示例，只提供任务要求。

例如：

```text
请总结这篇论文的研究问题、方法和实验结论。
```

优点：

* Prompt 较短；
* Token 消耗低；
* 编写成本低。

问题：

* 输出格式可能不稳定；
* 模型可能误解标准；
* 对复杂任务的控制力较弱。

---

# 8. Few-shot

Few-shot 是在 Prompt 中提供少量输入和输出示例。

例如：

```text
示例输入：
论文提出了一种新的图像分类方法……

示例输出：
研究问题：……
核心方法：……
实验结果：……
```

Few-shot 可以帮助模型理解：

* 输出格式；
* 任务边界；
* 判断标准；
* 特殊情况处理方式。

需要注意：

> Few-shot 示例的质量会直接影响模型输出。

如果示例包含错误、偏差或多余内容，模型可能模仿这些问题。

---

# 9. 正例、反例与边界示例

## 9.1 正例

正例告诉模型什么样的结果是合格的。

例如：

```text
正确输出：
论文报告模型在数据集 A 上获得 87.3% 的准确率。
```

---

## 9.2 反例

反例告诉模型哪些输出是不合格的。

例如：

```text
错误输出：
论文效果非常好。
```

问题在于：

* 缺少具体证据；
* 表述模糊；
* 无法验证。

---

## 9.3 边界示例

边界示例告诉模型在信息不足、信息冲突或无法判断时怎样处理。

例如：

```text
如果论文没有提供基线实验结果，请输出：

“原文未提供足够信息，无法判断该方法是否优于基线。”
```

边界示例可以降低模型随意推测的概率。

---

# 10. 目标、约束、边界与成功标准

## 10.1 目标

目标表示最终要完成什么。

例如：

```text
生成一份结构化论文阅读报告。
```

---

## 10.2 约束

约束表示执行过程中必须遵守的条件。

例如：

```text
只能使用提供的论文内容。
总字数不得超过 2000 字。
```

---

## 10.3 边界

边界表示哪些任务不属于模型当前的责任范围，或者哪些结论不能生成。

例如：

```text
不能判断未提供的实验数据。
不能声称论文结论已被后续研究证实。
```

边界不是简单的“不要做什么”，它用于规定模型能力和当前任务的责任范围。

---

## 10.4 成功标准

成功标准用于判断输出是否合格。

例如：

```text
研究问题、方法、实验、结果和局限五部分完整。
所有主要结论都能对应论文中的证据。
```

没有成功标准，就无法客观评价 Prompt 的效果。

---

# 11. 结构化输出

结构化输出是要求模型按照固定的数据结构返回结果。

## 11.1 通过示例约束结构

可以在 Prompt 中提供期望结构：

```json
{
  "research_question": "",
  "method": "",
  "results": "",
  "limitations": []
}
```

这种方式可以提高格式稳定性，但模型仍可能生成不合法的结构。

---

## 11.2 通过 Schema 或类定义结构

可以定义一个结构化输出类，并在 Agent 中设置：

```python
response_format = 类名
```

例如概念上定义：

```python
class PaperSummary:
    research_question: str
    method: str
    results: str
    limitations: list[str]
```

结构化输出主要保证：

* 字段存在；
* 类型正确；
* 程序能够解析。

但不能保证：

* 内容事实正确；
* 推理正确；
* 论文结论没有被误解。

需要记住：

```text
结构正确
≠ 内容正确
```

# 12. Prompt 模板

Prompt 模板用于将稳定规则与动态输入分离。

稳定部分包括：

* 身份；
* 目标；
* 执行规则；
* 输出结构；
* 质量标准。

动态部分包括：

* 论文正文；
* 用户问题；
* 用户笔记；
* 当前任务参数。

概念上：

```python
prompt = template.format(
    paper_content=paper_content,
    user_question=user_question,
)
```

Prompt 模板可以提高：

* 复用性；
* 一致性；
* 可维护性；
* 测试效率。

---

# 13. Prompt 版本管理

Prompt 不应写完一次后永久不变，而应像代码一样维护版本。

例如：

```text
paper_summary_v1
paper_summary_v2
paper_summary_v3
```

每次修改 Prompt 时需要记录：

* 修改了什么；
* 为什么修改；
* 使用了哪个模型；
* 使用了哪些测试论文；
* 哪些指标得到改善；
* 是否造成旧能力退化。

一个基本版本记录可以写成：

```markdown
## paper_summary_v2

### 修改内容

增加“证据不足时不得推测”的规则。

### 修改原因

V1 会在论文未提供实验数据时自行补充结论。

### 测试结果

在 5 篇证据不完整的论文中，错误推测数量减少。

### 新问题

部分正常回答变得过于保守。
```

# 14. Prompt A/B 测试

Prompt A/B 测试是在其他条件尽量相同的情况下，比较两个 Prompt 的效果。

需要保持一致：

* 模型；
* 输入论文；
* Temperature；
* 输出长度；
* 测试环境。

只修改 Prompt。

例如：

```text
Prompt A：简单 Prompt
Prompt B：结构化任务 Prompt
```

评价指标可以包括：

* 完整性；
* 事实准确率；
* 输出格式稳定性；
* 是否识别局限；
* Token 消耗；
* 响应时间。

A/B 测试的重点不是：

```text
哪个回答看起来更舒服
```

而是：

```text
哪一项 Prompt 修改导致了哪一项指标变化
```

---

# 15. Prompt 回归测试

Prompt 回归测试用于检查：

> 修改 Prompt 后，是否修复了新问题，却破坏了原来正常的能力。

例如，V2 增加：

```text
回答必须简洁。
```

可能产生：

* 输出长度降低；
* 但论文方法细节丢失。

因此需要维护固定测试集，例如：

```text
5 篇方法型论文
5 篇综述论文
3 篇实验型论文
2 篇证据不完整论文
```

每次修改 Prompt 后，都重新运行同一批测试。
