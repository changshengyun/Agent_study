# 3. MarkdownAndXML
在编写面向开发者与用户的提示消息时，可结合Markdown 格式与XML 标签，让 AI 模型清晰区分提示词、参考数据等内容的逻辑边界：

- Markdown 作用：通过标题、列表划分提示词不同板块，体现层级结构，同时方便开发人员阅读调试提示词。
- XML 作用：划定参考文档等独立内容的起止范围；还能通过标签属性给内容附加元数据，供指令调用引用。

开发者提示词通用固定板块（顺序可随模型微调）
- 身份设定：说明助手定位、沟通风格与核心目标；
- 执行指令：规定 AI 生成回复的全套规则，包含允许 / 禁止行为、自定义工具调用规范等，可按需拆分多个子模块；
- 示例样例：给出输入样例 + 对应的标准理想输出；
- 上下文资料：补充训练数据以外的私有、业务专属参考信息，通常放在提示词末尾，方便不同生成请求替换更换。

- 少样本学习：
小样本学习允许你通过在提示中包含少量输入/输出示例来引导大型语言模型完成新任务，而无需对模型进行微调。模型会隐式地从这些示例中“学习”模式并将其应用于提示。提供示例时，尽量展示各种可能的输入及其对应的预期输出。
- 原因：
（1）使模型能够访问专有数据，或模型训练所用数据集之外的任何其他数据。
（2）将模型的响应限制在您已确定的最有利的特定资源集上。

# 5. prompt优化策略
## 5.1 撰写清晰明确的指令
核心逻辑：模型无法猜测你的隐含意图，模糊的指令会带来不可控的输出

具体落地方法：
- 补充任务细节与约束条件，避免宽泛、模糊的表述
- 为模型设定专业角色与身份立场
- 使用分隔符（"""、###、XML 标签）清晰区分指令、上下文与输入数据
- 明确指定任务的执行步骤与先后顺序
- 通过少样本示例锚定输出标准
- 限定输出的格式、长度、语言风格与交付形态

## 5.2 提供参考文本降低幻觉
核心逻辑：用可靠的参考信息约束模型，减少编造内容的概率

具体落地方法：
- 添加正反示例：注入参考资料，要求答案必须基于给定内容生成
- 添加标注：引导模型标注信息来源、引用原文对应片段
- RAG：长文本场景的分段注入与结果汇总方法

## 5.3 拆分复杂任务为子任务
核心逻辑：分而治之，降低单步任务的复杂度与出错率

具体落地方法：
- 意图分类：先识别用户需求类型，再匹配对应处理逻辑
- 长文档处理：分步摘要、逐段处理后再整合最终结果
- 多环节任务：拆解为串行的单步调用流程，分步执行

## 5.4 给模型预留"思考"过程
核心逻辑：强制模型先推理再下结论，显著提升推理、计算类任务的准确率

具体落地方法：
- 思维链引导：要求模型分步写出完整推理过程
- 内心独白模式：隐藏推理过程，仅对外输出最终结果
- 自检机制：要求模型先自查疏漏与错误，再给出最终答案
- 慢思考引导：禁止模型仓促下结论

## 5.5 借助外部工具补全能力
核心逻辑：弥补模型自身的知识时效性、计算能力、专业深度短板

具体落地方法：
- 结合检索工具（RAG）补充专业领域与实时信息
- 调用代码解释器处理数学计算、数据处理、文件操作
- 函数调用（Function Calling）的提示词设计要点
- 多模态工具的配合使用方式

## 5.6 系统性测试与迭代
核心逻辑：用量化、可复现的方式验证提示词的优化效果

具体落地方法：
- 构建测试用例集，覆盖典型场景与边界场景
- 单一变量原则：每次仅修改提示词的一个要素
- 设定可量化的评估指标（准确率、格式达标率、合规性等）
- 维护黄金测试集，持续验证迭代效果
### prompt评估
项目program_output和最终助手message是两个独立的输出；请务必对两者都进行测试。理论上，程序可以返回正确的记录，但消息中可能缺少必填字段、引用或注意事项。

对相同的代表性任务，比较直接调用和程序调用。检查最终响应是否正确、完整，并包含所需的证据。然后比较总令牌数、延迟、成本、调用次数、轮次和重试次数。仅当响应仍然符合现有评估标准时，才将资源使用量的降低视为改进。
